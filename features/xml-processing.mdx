Operation 2 reads `ficheros-datos/operaciones_esa.xml` using Jackson's `XmlMapper`, then performs two sequential passes over the mission nodes: first it updates every mission row in the database, then it processes the crew roster and upserts any new crew members it finds.

## How to trigger

Start the application and enter `2` at the menu prompt:

```bash
Menú:
  1: Almacenar en CSV los tripulantes sin misiones asignadas.
  2: Procesar fichero operaciones_esa.xml
  3: Imprimir en consola el listado de misiones inciadas en un año.
----------------------------------------------------------
Introduce la operación a realizar:
2
```

The console confirms the start of processing:

```
Ejecutar procesamiento fichero operaciones_esa.xml
```

## XML file structure

The file `ficheros-datos/operaciones_esa.xml` is generated by ESOC Darmstadt. Its root element carries metadata attributes and contains a `<misiones>` block with one `<mision>` per operation.

```xml
<operacionesESA fecha_generacion="2040-11-12"
                centro_control_origen="ESOC Darmstadt"
                version_xml="3.0"
                md5="F1C92A88E23DF00A44AB9FF9982DE0AC">

    <misiones>
        <mision>
            <nombre>Reabastecimiento Ares I</nombre>
            <cod_nave>ESA-TRANS-02</cod_nave>
            <cuerpo_celeste>Marte</cuerpo_celeste>
            <fecha_inicio>2034-04-02</fecha_inicio>
            <fecha_fin_estimada>2034-04-25</fecha_fin_estimada>
            <estado>En curso</estado>
            <objetivo>Entrega ampliada de repuestos en Base Ares</objetivo>
            <presupuesto>18950000</presupuesto>

            <tripulacion>
                <tripulante ident_esa="2001">
                    <rol>Piloto</rol>
                    <nivel_estres>2</nivel_estres>
                </tripulante>
                <!-- additional tripulante nodes ... -->
            </tripulacion>
        </mision>

        <!-- additional mision nodes ... -->
    </misiones>
</operacionesESA>
```

### New crew member element

A `<tripulante>` node may optionally contain a `<nuevo_tripulante>` child. Its presence signals that the person must be inserted into (or updated in) the database:

```xml
<tripulante ident_esa="2101">
    <rol>Ingeniero</rol>
    <nivel_estres>2</nivel_estres>
    <nuevo_tripulante>
        <nombre>Elena</nombre>
        <apellidos>Markovic</apellidos>
        <pais>Serbia</pais>
        <nivel_experiencia>4</nivel_experiencia>
        <especialidad>Robótica Criogénica</especialidad>
    </nuevo_tripulante>
</tripulante>
```

The `ident_esa` attribute on `<tripulante>` is the ESA identifier used to look up the person in `T_TRIPULANTE`.

## How it works

### Parsing

`ProcesadorXMLMision.procesarFicheroXMLMisiones()` loads the file path from the `ruta.fichero.xml` property and parses the whole document into a `JsonNode` tree using Jackson's `XmlMapper`:

```java
File ficheroXML = new File(rutaFicheroXML);
ObjectMapper mapeadorXML = new XmlMapper();
JsonNode nodoOperaciones = mapeadorXML.readTree(ficheroXML);
JsonNode listaMisiones = nodoOperaciones.get("misiones").get("mision");
```

From this point forward all navigation is done with standard `JsonNode` traversal (`node.get("key")`, `node.asString()`, `node.asInt()`).

### Pass 1 — mission updates

<Steps>
  <Step title="Iterate over every &lt;mision&gt; node">
    The first `for` loop walks `listaMisiones` and calls `realizarTratamientoNodoMision(nodoMision)` for each one.
  </Step>

  <Step title="Build a Mision value object">
    The method extracts all scalar fields from the XML node and populates a `Mision` instance, including nested `CuerpoCeleste` (from `<cuerpo_celeste>`) and `Nave` (from `<cod_nave>`).
  </Step>

  <Step title="Update the database row">
    `IMisionDAO.actualizarMisionPorNaveCuerpoFecha(mision)` is called. The row is matched by the combination of `cod_nave`, `id_cuerpo_celeste` (resolved by name), and `fecha_inicio`. The following columns are updated:

    ```sql
    UPDATE T_MISION
    SET nombre              = ?,
        fecha_fin_estimado  = ?,
        fecha_fin           = ?,
        estado              = ?,
        objetivo_principal  = ?,
        presupuesto_asignado = ?
    WHERE cod_nave = ?
      AND id_cuerpo_celeste = (
          SELECT tcc.identificador
          FROM T_CUERPO_CELESTE tcc
          WHERE tcc.nombre = ?
      )
      AND fecha_inicio = ?
    ```
  </Step>
</Steps>

### Pass 2 — crew upsert

<Steps>
  <Step title="Iterate over every &lt;mision&gt; node again">
    A second `for` loop revisits each mission node to access its `<tripulacion>` child, then iterates over the `<tripulante>` nodes inside it.
  </Step>

  <Step title="Check for &lt;nuevo_tripulante&gt;">
    `realizarTratamientoNodoTripulante(nodoTripulante)` reads the `ident_esa` attribute and checks whether the node contains a `<nuevo_tripulante>` child. If the child is absent, the crew member is **skipped**.

    ```java
    int identificadorESA = nodoTripulante.get("ident_esa").asInt();
    JsonNode nodoNuevoTripulante = nodoTripulante.get("nuevo_tripulante");
    if (nodoNuevoTripulante != null) { ... }
    ```
  </Step>

  <Step title="Look up the ESA ID in the database">
    `ITripulanteDAO.obtenerTripulantePorIdESA(identificadorESA)` runs:

    ```sql
    SELECT *
    FROM T_TRIPULANTE tt
    WHERE tt.identificador_esa = ?
    ```

    The result determines the branch taken next.
  </Step>

  <Step title="Insert or update">
    | Result | Action |
    |---|---|
    | Record **found** | Populate `Tripulante` with the existing DB values (preserving name, surname, and join date), then call `ITripulanteDAO.actualizarTripulantePorIdESA(tripulante)` |
    | Record **not found** | Populate `Tripulante` from the XML node, set `fechaIngresoEsa = LocalDate.now()`, then call `ITripulanteDAO.insertarTripulante(tripulante)` |

    **Update SQL:**
    ```sql
    UPDATE T_TRIPULANTE
    SET nombre           = ?,
        apellidos        = ?,
        nivel_experiencia = ?,
        especialidad     = ?,
        fecha_ingreso_esa = ?
    WHERE identificador_esa = ?
    ```

    **Insert SQL:**
    ```sql
    INSERT INTO T_TRIPULANTE
        (nombre, apellidos, identificador_esa, nivel_experiencia, especialidad, fecha_ingreso_esa)
    VALUES (?, ?, ?, ?, ?, ?)
    ```
  </Step>
</Steps>

<Note>
  Crew members **without** a `<nuevo_tripulante>` element are listed in the XML for context but are not written to the database during this operation.
</Note>

<Info>
  When a crew member already exists in the database (record found), the update preserves the original `nombre`, `apellidos`, and `fecha_ingreso_esa` values from the database rather than overwriting them with the XML data.
</Info>

## Configuration

```properties
# Fichero XML Lectura
ruta.fichero.xml=ficheros-datos/operaciones_esa.xml
```

Change `ruta.fichero.xml` in `configuracion/aplicacion.properties` to point to a different XML file without modifying any Java source.

## Key classes

| Class / interface | Responsibility |
|---|---|
| `ProcesadorXMLMision` | Parses the XML and drives both processing passes |
| `IMisionDAO` | DAO contract; `actualizarMisionPorNaveCuerpoFecha(mision)` |
| `ITripulanteDAO` | DAO contract; lookup, insert, and update methods |
| `MisionDaoJDBC` | JDBC implementation of `IMisionDAO` |
| `TripulanteDaoJDBC` | JDBC implementation of `ITripulanteDAO` |
| `XmlMapper` (Jackson) | Deserialises the XML file into a `JsonNode` tree |
